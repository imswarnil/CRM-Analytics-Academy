# Bindings & Interactivity

**Bindings** make dashboards interactive. They let one widget's state — a selection, a filter, a result — feed into another query. This is how clicking a bar in one chart updates everything else.

## Two kinds of bindings

CRM Analytics evaluates bindings at two different times:

- **Selection bindings** — react in real time as a user clicks. Evaluated continuously.
- **Results bindings** — pull a value out of one query's results to parameterize another. Evaluated when queries run.

Most interactivity uses **selection bindings**.

## Anatomy of a binding

A binding is a function call embedded in a query's JSON. The most common is `cell()`, which reads a value from another query's result:

```text
"{{ cell(stage_filter.selection, 0, \"StageName\").asString() }}"
```

It says: take the selected row from `stage_filter`, read the `StageName` column, and return it as a string.

## Interaction and serialization functions

Bindings combine an **interaction function** with a **serialization function**:

- **Interaction**: `cell()`, `column()`, `selection()` — what to read.
- **Serialization**: `.asString()`, `.asObject()`, `.asGrouping()` — how to format it for the target query.

::note
Picking the right serialization function is the most common binding gotcha. A filter expects `.asObject()` or `.asString()`; a grouping expects `.asGrouping()`.
::

## A filter binding example

Suppose a list widget `region_list` lets users pick a region. A second query filters by that selection:

```sql
q = load "Opportunities";
q = filter q by 'Region' in {{ cell(region_list.selection, 0, "Region").asObject() }};
q = group q by 'StageName';
q = foreach q generate 'StageName' as 'Stage', sum('Amount') as 'Pipeline';
```

Now selecting a region instantly refilters the pipeline chart.

## Handling empty selections

When nothing is selected, a naive binding can break the query. Guard it with `coalesce`:

```text
{{ coalesce(cell(region_list.selection, 0, "Region").asObject(), "*").asObject() }}
```

::tip
Returning `"*"` (match-all) on empty selection keeps the dashboard showing everything until the user narrows it down — a much better default experience.
::

## Static vs. dynamic bindings

- **Static bindings** resolve once when the dashboard loads.
- **Dynamic bindings** re-evaluate on every interaction.

Use dynamic bindings for click-driven filtering; static for one-time configuration like a default date range.
